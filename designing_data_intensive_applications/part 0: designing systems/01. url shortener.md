# URL Shortener System Design

- URL shorterning service that converts long URLs into shorter, manageable links

## The Approach

1. Requirements
2. Core entities
3. API or Interface
4. Data flow [more to infra-heavy systems]
   - rate limiter
   - message queue
5. High-level design
6. Deep dives

## 1. Requirements | Functional vs. Non-Functional Requirements

1. We start with the core features: **functional requirements**

- top 2-3 features
- if you don't understand the system, ask questions

### Functional requirements -> Users should be able to/can... | What the system does

User can...

- Add a link to it and get a shorter version | create a short url from a long url
  - optionally:
    - support custom alias
    - support expiration time

- Be redirected to the original from the short url

### Non-functional requirements -> The system should be... | Qualities of the system [-ities] / How the system behaves

- From the `-ities`, which are interesting for this system?
- What's the scale of this system?
  - how many users?
    - e.g. 100 million daily users (DAU) and 1 billion urls that get shortened

The system should be...

- fast in redirecting [low latency / ~200ms - human perceivable]
  - less than 200ms, humans don't perceive -> 200ms = realtime
- scale to support 100M DAU & 1B urls
- ensure uniqueness of shortcodes (even with 1B urls!)
  - no collision!
- CAP
  - in the case of a URL shortener: availability > consistency
    - it's more important that the system is available and it's fine if the data might be stale for a while
    - high availability | eventual consistency

> Before doing some estimations, start you high-level design and if you need to make a further decision, then do it, based on the math

> Only do a calculation when the result of the calculations directly inform a direction

## 2. Entities and API

### Core entities: tables in the db

- Depending on the system, it's fine just to list the entities, then as you're doing the high-level do the data model

1. List the entities

- User
  - name
  - id [pk]
- Url
  - id [pk]
  - user_id [fk]
  - original_url
  - short_url
  - expiration_date
  - created_at

2. Think about the data model (as you're doing the high-level)

### API

- Contract between users and systems
- Most of the time, it's 1:1 mapping of a functional requirement and an API. So, that's why it's important to think well about the functional requirements

```ts
// METHOD /path -> [return]
// {
    // body
// }

// user can create a short link from a long URL (shorten url)
POST /urls -> short_url
{
    original_url,
    alias?,
    expiration_time
}

// user is redirected to original url from shortened one
GET /urls/:short_url -> original_url
```
