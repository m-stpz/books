# URL Shortener System Design

- URL shorterning service that converts long URLs into shorter, manageable links

## The Approach

1. Requirements
2. Core entities
3. API or Interface
4. Data flow [more to infra-heavy systems]
   - rate limiter
   - message queue
5. High-level design
6. Deep dives

## 1. Requirements | Functional vs. Non-Functional Requirements

1. We start with the core features: **functional requirements**

- top 2-3 features
- if you don't understand the system, ask questions

### Functional requirements -> Users should be able to/can... | What the system does

User can...

- Add a link to it and get a shorter version | create a short url from a long url
  - optionally:
    - support custom alias
    - support expiration time

- Be redirected to the original from the short url

### Non-functional requirements -> The system should be... | Qualities of the system [-ities] / How the system behaves

- From the `-ities`, which are interesting for this system?
- What's the scale of this system?
  - how many users?
    - e.g. 100 million daily users (DAU) and 1 billion urls that get shortened

The system should be...

- fast in redirecting [low latency / ~200ms - human perceivable]
  - less than 200ms, humans don't perceive -> 200ms = realtime
- scale to support 100M DAU & 1B urls
- ensure uniqueness of shortcodes (even with 1B urls!)
  - no collision!
- CAP
  - in the case of a URL shortener: availability > consistency
    - it's more important that the system is available and it's fine if the data might be stale for a while
    - high availability | eventual consistency

> Before doing some estimations, start you high-level design and if you need to make a further decision, then do it, based on the math

> Only do a calculation when the result of the calculations directly inform a direction

## Deep Dive - Non-functional requirements

- Avoid going too deep in any of the design decisions before having the whole system designed
- Once that's done, do the deep-dive

> "I'm black boxing this for now, but I'll come back to it later"

- Create short url
  - What do we need?
    - fast, unique, short (5-7 characters)

### How can we make it unique?

1. Prefix of the long url: bad idea

2. random number generator
   - how large would this number need to be
     - 0-9 = 10 nums + a-z = 24 lower case letters + A-Z = 24 upper case letters = 58^6 (6 length) = 38B -> base62 encoding
     - 62^6 = 56B
     - 1 billion = 10^9
   - what's the change of collision?
     - 880k out of each 1 billion
   - In our process, we can:
   1. generate random number (base62)
   2. check if this number is already taken (collision check)
      2.1. if yes, generate again
      2.2. if no, save it and return it

3. hash the long url
   - md5 (longUrl) -> hash -> base62[base][:6]
   - same as above

4. Counter
   - increment a counter -> base62
   - predictability which is bad for security
     - rate limiting
   - bijective function
     - sqids.org

### How can we make it low latency?

- Indexing: avoiding grabbing from disk to memory and targetting exactly where it's saved in disk
  - index = a pointer to where the entry lives in disk
  - Define a primary key
    - ensures: uniqueness and that an index is built for that table
      - postgres uses a b-tree for indexes (log n)

- If we don't want to go to disk, then we can add a cache
  - read through:
    - check cache first
    - if entry is there, return to server
    - else (cache miss), go to db, retrieve it, update the cache, and return it to server
  - LRU (least recently used) cache:
    - if cache gets full, delete the data that isn't so often used

```
                     cache // avoids going to disk
                    /     \
client <----> server <----> db
```

### How can we scale to support 100M DAU (Daily Active Users) and 1B URLs?

10^N = n is equal to number of zeros after 1

- 10^8 = 8 zeros after 1
- 10^3 = 3 zeros after 1

- 100M = 10^8 => 100.000.000
- around 100k (86400) seconds on a day (rounded up) = 10^5 = 100.000

```
10^8 / 10^5 = 10^3 (subtraction of exponents) = 1000 (requests per second -)
```
