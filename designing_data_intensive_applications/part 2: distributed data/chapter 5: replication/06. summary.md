# Summary

Replication can server several purposes:

- high availability: keeping the system running when one node or multiple are down
- partition tolerance: keeping the system working even though when there's network interruption
- latency: placing data geographically closer to users
- scalability: being able to handle a higher volume of reads than a single machine could do

Despite being a simple goal -- **keeping a single state of truth for data on several machines** --, it's quite tricky. It requires us to think about:

- concurrency writes
- data conflict

## Main approaches

- Single-leader replication: simple to understand and no conflict resolution to worry
  - clients send writes to a single node (leader), which sends a stream of data with the changes to the other replicas (followers)
  - reads can be performed on any replica, but reads from followers might be stale
- Multi-leader replication: more robust in case of faulty nodes, network interruptions, and latency spikes. However, increased complexity and eventual consistency
  - clients send writes to one of several leader nodes, any of which can accept writes
  - leaders send stream of data change events to each other and to the followers
- Leaderless replication
  - client send write to several nodes and read from several nodes in parallel in order to detect and correct nodes with stale data

## Sync vs. async replication

- Async replication is usually fast, however, we need to think about it when the lag increases
- If a leader is promoted and it was a async updated follower, data can be lost
- How a system should behave under replication lag
  - Read-after-write consistency: users should always see data they themselves submitted
  - Monotonic reads: after users have seen the data at one point in time, they shouldn't later see the data from some earlier point in time
  - Consistent prefix reads: users should see the data in a state that makes causal sense
