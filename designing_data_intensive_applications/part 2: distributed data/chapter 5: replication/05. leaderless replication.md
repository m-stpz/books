# Leaderless replication

- So far, in this chapter, we've seen single and multi-leader. This approach takes into consideration the client sending a write request to one node (leader), and the db system is responsible for propagating this write to the other replicas (followers/other leaders)
  - leader: determines the order of write
  - followers: apply the leader's writes in the same order

- Some data storage systems take a different apporach, they abandon the concept of a leader (a responsible node for writing), thus allow any replica to perform writes
  - this technique was mostly buried during the dominance era of relational databases
  - it started becoming more popular again when amazon used it to its in-house Dynamo system
  - riak, Cassandra, and Voldermort are open source datastores with leaderless replication models inspired by dynamo
  - this type of db is also known as dynamo-style

## Writing to the db when a node is down

- If you have a db with 3 replicas, and one of them is down, what happens?
  - leader-based: if the leader is the one down, you need to do a failover (promote a follower to a leader)
  - leaderless: the write is sent in parallel to all active nodes.
    - And the node down? It doesn't receive the write
      - once it gets back, you may get stale data from it
      - how to solve it?
        - read requests are sent to multiple nodes in parallel
        - version numbers are used to determine which value is newer
    - in a leaderless setup, failover doesn't exist

### Read repair and anti-entropy

- Replication system should ensure that eventually all the data is propagated
- Dynamo-style datastores use usually:

1. read repair
   - when a client makes a read request from several nodes in parallel, it can detect any stale responses
   - the client sees the stale value and writes the newer value back to that replica
   - this approach works well for frequently read values
2. anti-entropy process
   - background process to harmonize differences
   - copies any missing data to the replicas
   - doesn't copy writes in any particular order
   - without an anti-entropy process, not-frequently read values may e stale

### Quorums for reading and writing

If there are n replicas, every write must be confirmed by w nodes to be considered sucessfull, and we must query at least r nodes for each read

```
- n replicas
- w: confirmed writes on nodes
- r: queried nodes
w + r > n -> quorum condition | we get an up-to-date value when reading
```

- Reads and writes that obey the r and w values are called **quorum reads and writes**
- In dynamo-style dbs, the params n, w, and r are usually confi gurable
  - commonly n odd number (3/5)
  - math.floor(w = r = (n + 1) / 2)

#### Quorum condition `w + r > n`

- if w < n, we can still process writes if a node is unavailable
- if r < n, we can still process reads if a node is unavailable
- with n = 3, w = 2, r = 2, we can tolerate 1 unavailable node
- with n = 5, w = 3, r = 3, we can tolerate 2 unavailable nodes

#### Monitoring staleness

- From an operational perspecive, you should monitor whether your dbs are returning up-to-date results.
- You should monitor how replication is being propagated and whether it's healthy
- For a leader-based replication, the db typically exposes metrics for the replication lag, which you can feed into a monitoring system
- In leaderless replication systems, monitoring staleness is more difficult

## Sloppy quorums and hinted handoff

- Dbs with configured quorums can tolerate failure/slowness of indivial nodes without the need for failover
- They can return when w or r nodes have responded
- This makes dbs with leaderless replication great:
  - high availability
  - low latency
  - it's fine to have some stale data/read

## Detecting concurrent writes

- Events mayb arrive in different order at different nodes

```
client a -> set x = "hi"
client b -> set x = "hello"

node 1 -> receives x = "hi"
node 2 -> receives x = "hi", then x ="hello"
node 3 -> receives x = "hello", then x ="hi"

now, if you ask x, you get:
node 1 -> "hi"
node 2 -> "hello"
node 3 -> "hi"

What now?
```

- How do we deal with these write conflicts?

### Dealing with write conflicts

Approaches for achieving eventual convergen

1. Last write wins (discarding concurrent writes)

- each replica only needs to store the most "recent" value
- "older" values are overwritten and discarded
- we need a way to certaily define which write is more "recent" (this is non as obvious though)
  - we can attach a timestamp to each write, pick the biggest as the most "recenter" and discard the others
  - this is the LWW (last write wins)
- it achieves the eventual convergence, but at the cost of durability
  - only one write will survive
  - if losing data is acceptable, LWW is a good choice, otherwise stay away from it

#### The "happens-before" relationship and concurrency

- How do we decide whether two operations are concurrent or not?
  - An operation A happens before another operation B if it B knows/depends/builds about/on/upon A
  - If an operation doesn't happen before the other (neither knows about the other), they are concurrent
- there are 3 possibilities
  - A happens before B
  - B happens before A
  - A and B are concurrent
