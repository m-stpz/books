# Problems with replication lag

Replication gives us the ability of:

- tolerating node failures: if one goes down, another can go up
- scaling: handling more requests than one node can handle
- improving latency: placing replicas closer to users

> On leader-based replication, writes go through a single node (leader), but read-only queries go to any replica

For workloads that consisten of mostly reads and a few writes (common pattern for the web), there's an attractive option: **read-scaling architecture**

- Create many followers and distribute the read requests across them
- This removes load from the leader and allows read requests to be served by nearby replicas
- In this read-scaling architecture (creating many followers and distributing reading to them), it only works with async replication. Why?
  - if you tried async, only one node failing would cause:
    - system unavailable for writing
    - possibly data inconsistency

- However, with async, the follower can have fallen behind
  - if you run a query on the leader and in one follower, they might present different results
  - this state is only temporary, if we wait a little, it will catch up (eventual consistency)
  - there's no practical limit of how long the **eventual consistency** can be
    - in normal operation, the delay (between a write op on the leader and the replication to the follower) might by a fraction of a second, and not noticeable. This, however, could become multiple seconds, or even minutes (e.g. system operating near capacity/problem with the network)

Let's see some problems when this lag/delaygets too large, There are 3 possible problems that can happen when there's replication lag

## Reading your own writes

It's common for applications to allow the user to submit some data and then view what they've submitted

- the write goes to the leader, but a read could come from a follower
- if the replication lag takes too long, the user might think his data was lost (when in fact it wasn't. it's just taking a while to replicate)

How do we solve this?

- **read-after-write consistency** aka **read-your-writes consistency**
  - the writes of the user himself will be readily available
  - the users will always be able to see the updates they submitted themselves (other uses may take a while)
    - it ensures the user his own input has correctly been saved

How can we implement a read-after-write consistency in a leader-based system?

-**When reading something that the user may have modified, read it from the leader; otherwise, read it from the follower**

- for this, you need a mechanism to know something has been modified without querying it. Possibilities

1. if the field is only modifiable by the user (e.g. user profile information) always read it from the leader
   - any other users' profile from a follower

2. If most things are editable by the user in the application, the approach above won't work, since it'd defeat the purpose of the leader-follower architecture
   - in this case, other criteria might be used to check if we read from the leader

## Monotonic reads

Another thing that can happen from async replication is that users might see things moving back in time

- this can happen if user makes several reads from different replicas
- for instance, a user can access a page see some comments:

```
load 1
user 123 sees
comment abc by user 555
comment zzz by user 222

load 2 -> this "goes back" in time when abc wasn't present
user 123 sees
comment abc by user 222
```

- this can happen especially if the user refreshes the page and each request is routed to a random server and thus to a different follower

How to solve this?

- **monotonic reads**
  - if user reloads the page multiple times, they won't go back in time
  - they won't read older data after having read newer data
  - one way to do this is to ensure that each user always makes their reads from the same replica
    - this can be chosen based on the hash of the user id, rather than randomly

```
low                                             high
--------------------------------------------------> consistency
low                     monotonic               strong
consistency             reads                   consistency
```

## Consistent prefix reads

- What if two events are linked by causality, for instance:

A -> B

A user might read data from a replica where A hasn't been propagated, but B has, then the user would see:

B -> A

How do we solve this?

- **consistent prefix reads**
  - guarantees that if a sequence of writes has a specific order, users reading the data see they appear in the correct order
  - writes that are causally related to each other are written to the same partition

## Solutions for Replication Lag

When working with eventually consistent system, ask yourself

**How would the application behave if the replication lag increased to minutes or hours?**

- Don't pretend the replication is sync when it's not
- To reduce issues on replication issue, we can use **transactions**. They are a way of dbs providing stronger guarantees of replication
