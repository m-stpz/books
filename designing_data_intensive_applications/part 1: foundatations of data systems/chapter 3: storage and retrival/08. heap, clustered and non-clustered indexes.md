# Heap vs. Clustered Index vs. Non-Clustered Indexes

## Heap

Without a primary key or an index, data is saved in a Heap

- pages are just stacked together, without order
- new rows are just dumped into whatever page has free space
- no pointers linking pages in an ordered way
- to find anything, SQL must do a full table scan

## Clustered Index: Physical sort - usually the PK

once you define a primary key, SQL server automatically creates a Clustered Index

- with a primary key, the entire physical structure of the table changes, now it becomes a B-tree
- rows are physically sorted based on the index [like the `ID` column]
- usually, the primary key is the core of the clustered index, but you can use other portion of the data, such as the `createdAt`

> A table can only have only one clustered index because the data rows themselves can only by physically stored in one way

Think of a phone book, it's physically stored by last name. This is a clustered index.

> A table can have one clustered index, but multiple nonclustured

## Non-Clustered Index: Secondary lookups

- If we create a non-clustered index [e.g., indexing something beyond the primary key, like `firstName`], SQL builds a separate, mini B-tree just for that index
  - original data stays where it is: heap or clustered index
- Any other index that we create is a non-clustered index
- It doesn't rearrange the table physicaly in disk, it only creates a separate independent B-tree lookup structure

> Clustered is usually the PK, while the nonclustered is some **frequently used column**

- Differently from the clustered index, which at its leaves we find real data, for the non-clustered index, at the leaves, we point to where the data lives

| Attribute        | Clustered Index                                                             | Non-clustered Index                                                           |
| ---------------- | --------------------------------------------------------------------------- | ----------------------------------------------------------------------------- |
| Default creation | Automatically through the PK                                                | Manually for speeding up frequently queried columns                           |
| Physical storage | The index is the table. Data is physically stored in a sorted b-tree manner | A separate structure. Isolated lookup table that sits apart from the data     |
| Leaf content     | Actual data rows                                                            | Only the index column + pointer to where row lives in the real b-tree or heap |
| Limit per table  | only 1                                                                      | Multiple                                                                      |
