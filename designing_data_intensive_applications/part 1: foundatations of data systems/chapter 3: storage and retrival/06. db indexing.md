# Db indexing

- What's an index?
- What problem does it solve?
- What are the most common indexes?
- When should you use it?

- Dealing with disks (regardless if it's SSD or HDD) is more challenging then RAM, however, we accept this trade off since:
  - disks offer permanent/durable memory
  - lower cost per GB than RAM
    - as RAM memory increases, there are dbs that are opting for a "in-memory" approach

## What it solves

- Data is arranged in a db in pages (around 8kb of data ~ 100 rows)

Let's say we want to find a particular item in the db, without an index:

1. We grab "page 1" from the disk into the RAM (memory)

- The round trip from SSD to RAM is around 100ms
  - depending the size of our entries, we can even have queries that take 100s!

2. Scan all the rows
3. If we don't find it, we put it back to disk and grab the next one
4. Repeat steps 1-3 until we find it

> Indexes are data structures stored on disk

Searching an item with index:

1. Load the index memory (SSD -> RAM)
2. Check which page the index informs us the data is
   "Item x is on page 58"
3. Pull only that particular page

```ts
interface User {}

class UserTable {
  private diskRecords: User[] = []; // this is like our "disk" saved info
  private emailIndex = new Map<User["email"], DbTable["page"]>(); // maps an email to an array index (a page in the db)

  // insert
  public insert(user: User) {
    this.diskRecords.push(user); // push to disk

    const positionInTable = this.diskRecords.length - 1; // update the index
    this.emailIndex.set(user.email, positionInTable);
  }

  // slow search (no index used): O(n)
  public findByEmailSlow(email: User["email"]) {
    for (let i = 0; i < diskRecords.length; i++) {
      if (this.diskRecords[i].email === email) {
        return this.diskRecords[i];
      }
    }

    return null;
  }

  // fast search (index): O(1)
  public findEmailByIndex(email: User["email"]) {
    const position = this.emailIndex.get(email); // use the index to get the position

    if (!position) {
      return;
    }

    const record = this.diskRecords[position]; // jump directly to that memory location
    return record;
  }
}
```

## Types of indexing

### 1. B-trees: most popular db index

- basic tree structures
- each node in the tree is a sorted list of values with points to another page in disk
  - the child node can either be another pointer
  - or an actual page

```
          20 | 50 | 90
         /       \           \
  5 | 35 | 45  60 | 83 | 87    91 | 100 | 130
```

Searching with b-trees

1. Load root not into memory
2. Identify the index "range" of the given query
3. Pull this other index
4. Repeat step 2-3 until we get to a "leaf" node, a node where there's an actual page

### 2. Hash index: used for cache | limited in range and sorting

- Rarely used in production dbs
  - Even though for search of a given element both have a similar performance, B-trees **offer range queries and sorting**
  - usually used for in-memory stores (Redis) / caching

1.  Pass the query element (e.g. an email) to a hash function
2.  Hash function returns the key
3.  Then grab the value from that key, the value is a pointer to where the data exists on disk

### 3. Geospatial | 2-d data

- This is a type of data that B-trees aren't that effective
  - B-trees excel at **1 dimensional data**, but not **2-D**

- Geospatial indexing:
  - geohashing
    - very popular
    - it's an "algorithm" that breaks areas recursively into a "deeper" more precise coordinate point, then it "feeds" into a B-tree
  - quadtrees
    - not used in production that much nowadays
  - R-trees

## Bottleneck optimizations

- If we want to improve performance, we can follow:

1. Database indexing
2. Read replicas (master-slave replication to offload reads)
3. Caching layer: in-memory db (e.g. Redis)

### In memory dbs

- Reads are served entirely from memory
- When an in-memory db is restarted, it needs to reload its state (either from disk, or over the network from a replica)

- Options:
  - redis
  - RAMCloud
  - VoltDB
  - MemSQL
