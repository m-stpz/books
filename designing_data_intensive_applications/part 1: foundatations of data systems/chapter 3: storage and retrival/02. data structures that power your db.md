# Data structures that power your db

- Consider the world's simplest db, implemented in TS
  - A basic key/value storage system

```ts
import * as fs from "fs";

class TinyDB<T> {
  // O(1) look time with Map
  private data: Map<string, T>;
  private filePath: string;

  constructor(filePath: string) {
    this.filePath = filePath;

    // load existing data or start fresh
    if (fs.existsSync(filePath)) {
      const raw = fs.readFileSync(filePath, "utf-8");
      this.data = new Map(Object.entries(JSON.parse(raw)));
    } else {
      this.data = new Map();
    }
  }

  // save the current state to the disk
  private save() {
    const obj = Object.fromEntries(this.data);
    // serialize the whole map into a JSON string and overwrite the local file
    fs.writeFileSync(this.filePath, JSON.stringify(obj, null, 2));
  }

  set(key: string, value: T): void {
    this.data.set(key, value);
    this.save(); // save the result on mutation
  }

  get(key: string): T | undefined {
    return this.data.get(key);
  }

  delete(key: string): boolean {
    const result = this.data.delete(key);
    this.save(); // save the result on mutation
    return result;
  }
}

interface User {
  //...
}

const db = new TinyDB<User>(".database.json");

db.set("user_1", { name: "Alice", isCool: true });
console.log(db.get("user_1"));
```

- Appending something to a file is usually rather efficient
  - This is log

> Even though log is used to refer to application logs (where application outputs text to describe what's going on), in this context it means: an append-only sequence of records (human-readable or not)

- This is the core principles of dbs
  - of course, with much more xomplex logic/controls for:
    - concurrency control
    - reclaiming disk space
    - handling error
    - handling partially written records

- Our db has a map, instead of an array.
  - Look up O(1) instead of O(n)
  - Indexing allows efficient data storage
    - indexing: keeping additional metadata on the side
      - acts as a sign post and helps locating the data more easily (e.g. key-value pair)
    - For writes, it's difficult simply appending to the end of some data structure. Some, any kind of indexing will slow writing down, however, it can dramatically increase read-time

```ts
// append-only store example
import * as fs from "fs";

class SimpleAppendOnlyStore {
  private filePath = "./database.log";

  // writing: just append to the end of the file. super fast
  public write(key: string, value: string) {
    const entry = JSON.stringify({ key, value }) + "\n";
    fs.appendFileSync(this.filePath, entry);
  }

  // scan from the bottom to top to find the newest version
  public read(key: string) {
    const lines = fs.readFileSync(this.filePath, "utf-8").split("\n").reverse();

    for (const line of lines) {
      if (!line) {
        continue;
      }

      const data = JSON.parse(line);

      if (data.key === key) {
        return data.value;
      }
    }

    return null;
  }
}
```

- In order to efficiently find the value for a particular key in the db, we need a new data structure, an index
- This chapter is about looking at different index strategies and seeing how they compare
  - they keep some metadata on the side, which acts a signpot and helps us to locate the data we want
  - index = additional structure that's derived from the primary data
  - maintaining additional structures incurs overhead, specially on write
  - any kind of index slows down write, because the index must also be updated every time data is written

> Important trade-off in storage systems: (well-chosen) indexes speed up read queries, at the cost of the write speed

- This is the reason that dbs don't index everything by default, they require you, the application developer, to choose indexes manually, using your knowledge of the application's typical query patterns
