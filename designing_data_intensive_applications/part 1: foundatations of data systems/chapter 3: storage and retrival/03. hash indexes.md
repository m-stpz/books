# Hash indexes

- Key-value stores are quite similar to the dictionary/object type in many languages
- The simplest index strategy is: keep an in-memory hash map of where every key is mapped to a byte offset in the data file

| key     | byte offset                                                           |
| ------- | --------------------------------------------------------------------- |
| value 1 | 0 // this is the range where this value [e.g. an object] lives (0-64) |
| value 2 | 64                                                                    |

- Given that we only append to a file, how do we avoid running out of space?
  - Since, in our current conceptualization, we're just appending, how do we deal with repeated values?
    - Break the log into segments of a certain size by closing the segment once it achieves a certain size. This is called compaction
    - I allocate "5 sizes" for this segment

```ts
interface LogEntry {
  key: string;
  value: any;
}

function compact(segments: LogEntry[]): LogEntry[] {
  const latestValues = new Map<string, any>(); // initialize an empty map to store the keys we've seen while moving "backwards"

  // iterate from most recent to oldest
  for (let i = segments.length - 1; i >= 0; i--) {
    const entry = segments[i];

    /* 
        most recent entry 10 => we only care about this one
                ... the others we eliminate
                    entry 5
                    entry 4
        oldest      entry 1
    */

    // if we haven't seen this key yet, it's the most recent one (only add the first seen key -> the others are already "stale")
    if (!latestValues.has(entry.key)) {
      latestValues.set(entry.key, entry.value);
    }
  }

  // convert the map into an array of entries
  // these are unique, meaning we compacted the entries
  return Array.from(latestValues.entries()).map(([key, value]) => ({
    key,
    value,
  }));
}

const logHistory: LogEntry[] = [
  { key: "user_1", value: { name: "Alice", status: "online" } }, // Entry 0 (Oldest)
  { key: "user_2", value: { name: "Bob", status: "offline" } }, // Entry 1
  { key: "user_1", value: { name: "Alice", status: "away" } }, // Entry 2 (Update)
  { key: "user_3", value: { name: "Charlie", status: "online" } }, // Entry 3
  { key: "user_1", value: { name: "Alicia", status: "active" } }, // Entry 4 (Final Update)
];

console.log(compact(logHistory));
```

## Implementation details

- File format: CSV is not the best format for a log
  - Faster and simpler to use binary format that first encodes the length of a string into bytes, followed by the raw string

- Deleting records
  - If you want to delete a key and its associated value, you have to append a special deletion record to the data file
  - When log segments are merged, this deletion record (tombstone) tells the merging process to discard any previous values for the deleted key

```
item 1 value: 1
item 1 value: 5
item 1 value: 10
item 1 deleted (tombstone)
merge process (identifies the tombstone - don't "merge" entries with item 1)
```

- Crash recovery
  - It the db is restarted, the in-memory hash maps are lost
  - In principle, you could restore the segment by reading the entire segment file from beginning to end
  - It can store snapshots of each segment's hash map on disk, which can be loaded into memory more quickly

- Partially written records
  - Dbs may crash at any time, including halfway through appending a record to the log
  - You can have some algo (checksums) that identifies partially written ones and remove/ignore them

- Concurrency control
  - One common choice is to have only one writer thread, this way multiple threads don't fight for supremacy
  - Write: one thread | Read: multiple

## Pros and cons of append-only logs

### Pros

- Sequential operations are quite effective and fast (much faster than random writes)
  - Both in hard-disks as well as SSDs
- Concurrency and crash recover is much simpler
- Merging old segments avoid the issue of data getting fragmented overtime

### Cons

- Hash table must fit in memory, so large number of keys might be an issue
  - Also hash collisions are a thing
- Range queries aren't efficient
