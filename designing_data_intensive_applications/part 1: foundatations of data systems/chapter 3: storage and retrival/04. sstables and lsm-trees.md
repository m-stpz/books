# SSTables and LSM-Trees

- We can improve our log strategy (appending at the end) by sorting the key-value pair by key
  - This format is: Sorted String Table, or SSTable
- In this format, we can't append the new key-value pairs to the segment immediately, since writes can occur in any order
- SSTables (Sorted String Stable) is one of the most important data structure in modern high-performance dbs like Cassandra, RocksDB, and BigTable
  - SSTable = sorted, read-only file on the hard drive that maps keys to values

## Log evolution

1. Simple log: just append everything

- Writes are fast, reads are slow
- disk space is wasted

2. SSTable: sort the data before writing

- Writes are slightly slower, but reads become fast

3. LSM-Tree: Use the "memtable" and multiple "SSTable" layers

- LevelDB and Cassandra use this

## How SSTables work

1. Sorted: every key inside the file is in alphabetical/numerical order
2. String: the keys and values are just byte arrays (strings)
3. Table: it's a mapping of key -> value

- SSTable (Sorted string table), it's a structured file on disk, rathen than a "live" object in memory
  - If we were to open it, it'd look like a sorted list of key-value pairs
  - even if you have 100gb of data, you can find a specific key by reading only a few kilobytes of index and data
  - since it's sorted, you never wander aimlessly

### What a SSTable looks like

Looks like a log file or a dictionary

- SSTable -> flat file optimized for sequential access

```
[Data block 1]

- key: "apple", value: "1.50"
- key "banana", value: "2.00"

[Data block 2]

- key: "cherry", value: "2.40"
- key: "tomato", value: "1.80"

[Footer/index]

"apple" is at byte 0
"cherry" is at byte 512
```

SSTable has:

1. Data block: keys and values in alphabetical order
2. Index block: "summary" that maps the key to the exact byte offset in the file

### Memtable

- Memtable: in-memory buffer that acts as the "waiting room" for the data before it's written to permanent storage on disk
  - sorted, in-memory cache
  - turns random write requests into organized, sequential files on disk
- Solves the biggest bottleneck in computer science:
  - Disk I/O is slow, but RAM is fast
    - Fast write: writing to RAM is basically instanteneuos
      - writing to disk is slow
    - Sequential disk usage: sequential writes are the fastest way to move data to disk
      - Sort, let it "fill", once it's full, flush to disk sequentially
    - Instand reads

- When we write data to some dbs, it doesn't go straight to the disk (too slow), instead it goes to the memtable
  1. the write: data is inserted into memtable
  2. the sorting: memtable is a sorted data structure (like the binary search tree)
  - no matter the arrival order, the data is kept sorted
  3. the flush: once the memtable reaches a certain size, it's frozen and "flushed" to the disk
  - since the data is already sorted, the db can write it as one continuous, sequential block
  - this created file is called an SSTable

## Tradeoff for SSTables

- Write is fast, read is slower
  - It's easy to keep a sequential write, however, checking the most recent entry from a given item is more expensive

## Constructing and maintaining SSTables

- How do we get our data to be sorted by key? Incoming writes can come in any order
  - Red-black trees
  - AVL trees
    - They allow us to insert keys in any order and read them back in sorted order

- Simplest BST (Binary Search Tree - Sorted memory structure)

```ts
class Node {
  constructor(
    public key: string,
    public value: any,
    public left: Node | null = null,
    public right: Node | null = null,
  ) {}
}

class SimpleMemtable {
  private root: Node | null = null;

  // 1. insert
  public put(key: string, value: any) {
    this.root = this._insert(this.root, key, value);
  }

  private _insert(node: Node | null, key: string, value: any) {
    if (!node) {
      return new Node(key, value);
    }

    if (key < node.key) {
      node.left = this._insert(node.left, key, value);
    } else if (key > node.key) {
      node.right = this._insert(node.right, key, value);
    } else {
      node.vale = value; // update if key exists
    }

    return node;
  }

  // in-order traversal
  // visits the nodes in alphabetical order
  public getSortedData() {
    const result: any[] = [];

    this._inOrder(this.root, result);

    return result;
  }

  private _inOrder(node: Node | null, result: any[]) {
    if (!node) {
      return;
    }

    this._inOrder(node.left, result); // visit left
    result.push({ key: node.key, value: node.value }); // visit self
    this._inOrder(node.right, result); // visit right
  }
}
```

### Strategy for engine store with balanced tree

1. When a write comes in, add it to an in-memory balanced tree data structure (e.g, red-black tree)

- This in-memory tree is called a memtable

2. When the memtable gets bigger than a threshold (few megabytes), write it out (flush) to disk as a SSTable file

3. To serve a read request:
   3.1 try on the memtable
   3.2 on the most recent on-disk segment
   3.3 next-older segment...

4. From time to time, merge and compact

- discard overwritten or deleted values

The approach above works well, but if the db crashes, the most recent writes are lost

- values that are in the memtable, but not yet on the SSTable
- to solve it:
  - keep a separate log on disk that saves each write, not in alphabetical order, just appended
  - its only purpose is to restore a memtable after a crash
  - everytime a memtable is written out to an SSTable, the corresponding log can be deleted

> The terms SSTable and Memtable were introduced by Google's Bigtable paper

- This indexing structure was described under the name Log-Structured Merge-Tree (LSM-tree)
  - Keeping a cascade of SSTable that are merged in the background
  - Can support an extremely high write throughput
- Storage engines that rely on the principle of merging and compacting sorted files are called LSM storage engines
