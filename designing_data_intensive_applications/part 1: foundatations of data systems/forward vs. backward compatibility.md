# Forward vs. Backward Compatibility

- In software architecture, API design, and data serialization, managing change is one of the hardest problems you'll face
- Systems evolve, but components are rarely upgraded at the exame same moment
- To prevent updates from breaking your system, we need to design for two distinct types of compatibility: forward and backward

## Backward compatibility | Looking to the past (new reads old)

- A new system can read or process data generated by an old system
- New code must still understand the legacy `___` that your users or older services are still using
  - formats
  - databases
  - API payloads

## Forward compatibility | Looking to the future (old handles new)

- Old systems can handle/ignore data generated by new system without crashing
- Harder to achieve, since you need to anticipate changes you haven't made yet
- Old system doesn't need to understand the new feature, only not blow up when it encounters them

| Aspect                 | Backward Compatibility                       | Forward Compatibility                                    |
| ---------------------- | -------------------------------------------- | -------------------------------------------------------- |
| Who is reading?        | New version                                  | Old version                                              |
| What are they reading? | Old data/formats                             | New data/formats                                         |
| What's the goal?       | Support legacy formats                       | Not to crash with new values                             |
| The ease of design     | High (you know what the old data looks like) | Low (requires strict rule on how to handle unknown data) |
