# Relational model versus document model

- SQL is the best-known data model [proposed by Edgar Codd in 1970]
  - Data is organized into relations (tables in SQL)
  - Each relation is an unordered collection of tuples (rows in SQL)

- By mid-1980s relational db managament systems (RDBMS) and SQL had become the default tool for storing and querying data with some sort of structure
- The goal of relational model was to hide the implementation detail (the internal representation of the data in the db) behind a clean interface
- The original purpose of relational dbs lied in **business data processing**
  - Transaction processing: sales, bank transactions, airline reservations, stock management
  - Batch processing: customer invoicing, payroll, reporting

## The birth of NoSQL

- NoSQL (2010s) is the last attempt to overthrow relational model's dominance
- NoSQL was not the intended name for distributed, nonrelational dbs (it came from a catchy twitter hashtag for a open source meetup)
- It has been retroactively interpreted as Not Only SQL
- Why its adoption?
  - Greater need for scalability (very large datasets or very high write throughput)
  - Preference for free and open source software over commercial db products
  - Specialized query operations that aren't well supported by the relational model
  - Frustation with restrictiveness of relational schemas
  - Desire for a more dynamic, expressive data model

## The Object-Relational Mismatch

- Most application development today is done in object-oriented programming, which leads to a criticism of the SQL data model
- If data is stored in relational tables, it's necessary a(n) (akward) translation layer between the objects in the application code and the db model of tables, rows, and columns [this is called impedance mismatch]
- Object-relational model (ORMs) can help reducing boilerplate, but they don't hide/eliminate the differences
- As the data grows, storing and retrieving the data in relation model can become more expensive (and unnecessarily weird, e.g. using a bunch of joins to grab some info). That's why some developers prefer "json", since it feels more aligned with the object-model and it's not necessary so many queries to grab info, since the structure is present on the first query

## Many-to-one and many-to-many relationships

- In fields such as `region_id` and `industry_id`, instead of storing a string, there are advantages to have a standardized list of "available elements"
  - Consistent style and spelling across profiles
  - Avoiding ambiguity
  - Easy of updating
  - Localization support
  - Better search

- Advantage of using ID, which has no meaning to humans, is that it never needs to change
  - The ID can remain the same, even if the info it identifies changes
  - Anything that is meaningful to humans may need to change sometime in the future

> If you're duplicating values that could be stored in one place, the schema is not normalized

- If the db doesn't accept joins, we compensante this in the application code through multiple queries
  - the work of making the join shifts from the db to the application code
  - even if the initial version of an applicatoin fits well in a join-free document model, data has the tendency to become more interconnected as features grow

### Relations

#### One-to-one:

- one record in table A relates exactly to one record in table B

```txt
|users (table a) | | identity_cards (table b) |
| john doe | -> | 123abc|
```

- How the join works: link the `user_id` on the card table to the `id` on the user table
  - the "secondary" table has an `<entity>_id` that links to the `id` on the primary table

#### One-to-many

- one record in table A relates with multiple records on table B
- one user has many posts/books/notes, etc
- most common relationship

- How the join works: the "many" side has a FK that points to the "one" side

#### Many-to-many

- one record in table A relates to multiple records on table B and vice-versa
- one user has many playlists, but many playlists are favorited by many users
- how the join works: relational dbs can't do this directly, then we need a third table "join table" `UsersPlaylists` that holds the ids of both

### Database normalization

- Process of removing duplicated information in your db
- Every information should be stored only once
- If you have to update some field in different tables, your schema isn't normalized
- To normalize data, we need _many-to-one_ relationships (e.g, many people live in one particular region, favorited the same song, etc), which doesn't fit nicely into document model
