# Graph-Like Data Models

- Does you application have many one-to-many relationships? [tree-structured data]
- Does it have some relationship between records?
  - If the answer to these questions is no: document model is appropriate
  - If the answer is no: then perhaps document model isn't appropriate

- What if many-to-many relationships are very common?
  - Relational model can handle simple cases of many-to-many
  - However, as the connections get more complex, a graph data model becomes more suitable

- A graph has:
  - vertices/nodes/entities
  - edges/connections/relationships

- Examples:
  - social graphs
  - web graph
  - rail network

## Property graphs

- In this model...

...each node consists of:

- unique identifier
- set of outgoing edges
- set of incoming edges
- collection of properties (key-value pairs)

...each edge consists of:

- unique identifier
- node at which the edge starts (the tail node)
- node at which the edge ends (the head node)
- label to describe the relationship between the two nodes
- a collection of properties (key-value pairs)

- We can think of a graph store consisting of two relational tables
  - one for nodes
  - one for edges

```sql
CREATE TABLE nodes {
    node_id integer PRIMARY KEY,
    properties json
};

CREATE TABLE edges {
    edge_id interger PRIMARY KEY,
    origin_node integer REFERENCES nodes (node_id),
    destination_node integer REFERENCES nodes (node_id),
    label text,
    properties json
};

CREATE INDEX origin_edges ON edges (origin_nodes);
CREATE INDEX destination_edges on edges (destination_nodes);
```
