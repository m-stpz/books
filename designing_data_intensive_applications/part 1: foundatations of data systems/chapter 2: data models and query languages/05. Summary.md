# Summary

- Initially, data was represented hierarchically (tree model), however, since it wasn't good to represent many-to-many relationships, the relational model was created
- New nonrelational "NoSQL" datastores have diverged:
  - document dbs: data comes in self-contained chunks
    - useful for messy, flexible data
      - schema-less
      - relational dbs need migration
    - data that changes often
    - data that must be scalable
  - graph dbs: targets scenarios where everything is possibly related to everything
    - useful for crime detection, recommendation systems

- Neither document nor graph enforce a schema on the data they store
  - makes it easier to adapt data to changing requirements
  - the data will certainly have some assumption of the data format
    - explicit (enforced on write - relational)
    - implicit (assumed on read - document/graph)

- Relational db
  - schema on write. The schema must conform on write to the structure
  - structure is explicit
  - scales "up" well (vertical)
  - strong structure
- Document db
  - scheme is on read. The schema is flexible
  - structure is implicit
  - scales "out" well (horizontal)
  - reads are incredibly fast
