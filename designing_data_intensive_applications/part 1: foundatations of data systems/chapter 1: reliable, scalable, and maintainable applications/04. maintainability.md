# Maintainability

- Majority of cost of software, it's not initial development, but ongoing maintenance
  - Fixing bugs
  - Keeping its systems operational
  - Investigating failures
  - Adapting it to new platforms
  - Modifying it for new use cases
  - Repaying technical debt
  - Adding new features

- Software should be designed in such a way that it minimizes maintenance pain
- There design principles for software systems

1. **Operability**: make it easy for operations teams to keep the system running smoothly
2. **Simplicity**: make it easy for new engineers to understand the system, by removing as much complexity as possible from the system
3. **Evolvability**: Make it easy for engineers to make changes to the system in the future
   - Adapting it for unanticipated use cases as requirements change
   - AKA: extensibility, modifiability, plasticity

## Operability: Making life easy for operations

> Good operations can work around the limitations of bad (or incomplete) software, but good software cannot run reliably with bad operations

- Operations is fundamental to keep a software system running smoothly
- Responsible for making routine tasks easy, allowing ops team to focus effort on high-value activities

### Responsibilities

- Monitoring, problem-causing, up-to-date software, anticipating problems, good practices, complex maintenance, defining process, org's knowledge

- Monitoring system health and capacity to restore service if it goes to bad state
- Tracking down cause of problems
  - System failures
  - Degraded performance
- Keeping software and platform up to date, including security patches
- Keeping tabs on how different systems affect each other
  - Problematic change can be avoided before it causes damage
- Anticipating future problems and solving them before they occur (e.g. capacity planning)
- Establishing good practices and tools for deployment, config management
- Performing complex maintenance tasks (e.g. moving application from one platform to another)
- Maintaining the security of the systems as config changes
- Defining processes to make operations predictable and help keep the prod env stable
- Preserving org's knowledge about the system, even as individuals come and go

### Making routine tasks easy

- Monitoring, automation, documentation, default behavior, self-healing, predictable behavior

- Providing visibility into the runtime behavior and internals of the system with good monitoring
- Providing good support for automation and integration with standard tools
- Avoiding deps on individual machines
  - allowing machines to be taken down for maintenance while the system as a whole continues running
- Providing good documentation and an easy-to-understand operational model
  - if I do x, y will happen
- Providing good default behavior, which can be, on demand, overriden
- Self-healing, when appropriate, with also admins manual control over the system state when needed
- Exhibiting predictable behavior, minimizing surprises

## Simplicity: Managing complexity

> A software mired in complexity is sometimes described as "big ball of mud"

- large complexity generates
  - system harder to reason about
  - more and varied hidden assumption, unintented consequences, unexpected interactions

- Accidental complexity: complexity that comes from the implementation, not inherit in the problem the software solves
- Abstraction helps in removing accidental complexity
  - A good abstraction can hide implementation details behind a clean, simple to understand interface

### Abstractions

- High-level programming languages are abstractions that hide machine code, CPU registers, and syscalls
  - We're still using machine code, however, not directly
- SQL is an abstraction that hides complex on-disk and in-memory data structutes, concurrent requests from other clients, and inconsistencies after crashes

### Symptoms of complexity

- Explosition of state space
- Tight coupling of modules
- Tangled dependencies
- Inconsistent naming and terminology
- Hacks
- Special-casing to work around issues elsewhere
