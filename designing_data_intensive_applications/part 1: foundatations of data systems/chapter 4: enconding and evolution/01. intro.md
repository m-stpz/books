# Intro to Encoding and Evolution

- Evolvability is important. We should make systems that make it easy to adapt to change
- Usually as the requirements evolve, it causes the data to be changed
  - new field
  - new record type

- Different data models deal with this differently
  - Relational: assumes the data conforms to to a schema
    - this schema needs to be changed [migrations]
  - Document: don't enforce a schema, so the db can have a mixture of legacy and newer data formats

- When data format/schema changes, usually a change in the application code is necessary
  - add a new field to the schema, then the application code reads that new field and write to it
  - in a large application, code changes often cannot happen instantaneously
    - rolling upgrade (deploying a new version to a few nodes at a time)

- This means that old and new versions of the code, as well as new and old versions of the data format, may coexist at the same time
- This way, we need to maintain **compatibility** in both directions

1. Backward compatibility: new version deals well with the previous version (newer code can read data that was written by old code)
2. Forward compatibility: previous version deals well with new version (older code can read data that was written by new code)

In this chapter we'll look at:

- several formats for encoding data (JSON, XML, Protocol Buffers, Thrift, and Avro)
- how they handle schema changes
- how they support systems where old and new data/code need to coexist
- how these formats are used for data storage and communication (web services, REST, RPC, and message-passing systems [message queues and actors])
