# Formats for encoding data

- Programs need to deal with data in (at least) two representations

1. In memory: objects, arrays, structs, lists, trees, hash tables, etc
   - optimized for efficient access and manipulation by the CPU (typically using pointers)
2. when we want to write data to a file or send it over the network
   - we want to encode it as a self-contained sequence of bytes
   - e.g. JSON document
   - this sequence of bytes is normally quite different from the data structures used in memory

Given the difference of in-memory vs. over-the-network or writing-mode, we need a translation between them

> How a program thinks about data (in-memory) to how it shares/saves data (sequential bytes) is a fundamental distinction

- This translation, from the in-memory representation to a byte sequence, is called encoding (aka, serialization or marshalling)
- The reverse is decoding (parsing, deserialization, unmarshalling)

Encoding: in-memory -> byte sequence

- It's similar to having an assembled shelf in the living room, dismantling it into a flat-box that is easy to ship

Decoding: byte sequence -> in-memory

- Once the person receives the box, it follows the instructions, and rebuilds the bookshelf

## Why can't we use db migrations?

- For 'sitting' data, data that is in the db, definitely, we can migrate the data to ensure that older versions update themselves according to the new schema
- However, this is only one portion of the issue. Data isn't only sitting on the db, it's also:
  - in-flight over the network
  - frozen in immutable logs

- When data is moving, traditional db-style migrations break down for 3 reasons

### 1. "Data in flight" problem | Network APIs

If you have a frontend, mobile app talking to a backend API, or a microservice A talking to a microservice B, we can't run a migration on the network pipeline

```
[Mobile APP V1] --- Sends old payload ---> [Backend server V2] (must parse v1 bytes)
```

- No sync upgrades: we wan't force users on app store to update their mobile app at the exact second we deploy the server
- Backend must be forward and backward compatible because it'll be handling requests from V1, V2 and V3 apps at the same time

### 2. "Rolling deployment" window

- If we're just updating the backend servers, deployments aren't instanteneous
- If we have 10 instances of a service running behind a load balancer, a deployment replaces them one by one

```
              ----> [Server 1: running v1 code]
Load balancer ----> [Server 2: running v1 code]
              ----> [Server 3: running v2 code] <-- just deployed

```

- During this deployment windows, v1 and v2 servers are running side-by-side
- If encoding format isn't evolvable, the cluster will start throwing serialization errors and dropping traffic midway through deploy

### 3. The "big data" problem | Immutable logs

What about systems that use append-only logs [Apache kafka], or data lakes where raw event data is dumped into cloud storage for analytics?

- Once the data is too large, we can't run a migration to change 3 years of historical files
- Computationally too expensive
- Logs are immutable records of history

Therefore, analytics code must be able to decode bytes from all its versions

## Language-specific formats

Many programming languages have built-in support for encoding in-memory objects into byte sequence

- Even though this comes built-in in the language, it's generally a bad idea to use your language's built-in encoding for more permanent functionality
  - built-in encoding causes:
    - tight coupling for encoding/decoding purposes for that specific language
    - security and performance issues
    - data incompatibility, both forwards and backwards

### JS

#### JSON

```js
const userSession = {
  username: "Jon",
  loginTime: new Date(),
  roles: ["editor"],
  getGreeting() {
    return `hello, ${this.username}`;
  },
};

// encoding: in-memory -> byte sequence
const encodedData = JSON.stringify(userSession);
// output: {"username":"Jon","loginTime":"2026-05-22T17:35:14.247Z","roles":["editor"]}

// decoding: byte sequence -> in memory
const decoded = JSON.parse(encodedData);
```

#### Protobuf

1. The schema `user.proto`

- Single source of truth for sender and receiver

```ts
syntax = "proto3";

message UserProfile {
  int32 id = 1;
  string name = 2;
  repeated string skills = 3; // 'repeated' means an array/list
  bool is_active = 4;
}
```

2. JS encoding

- Load the schema
- Use it to validate the in-memory object
- Compile it down to a binary

```js
const protobuf = require("protobufjs");

// In a real app, you would load the .proto file.
// For this example, we'll parse the schema string directly.
const root = protobuf.parse(`
  syntax = "proto3";
  message UserProfile {
    int32 id = 1;
    string name = 2;
    repeated string skills = 3;
    bool is_active = 4;
  }
`).root;

// Get the message type from our schema
const UserProfile = root.lookupType("UserProfile");

// ==========================================
// 1. IN-MEMORY REPRESENTATION
// ==========================================
const userProfile = {
  id: 42,
  name: "Alex",
  skills: ["JavaScript", "TypeScript"],
  isActive: true, // JavaScript camelCase maps to proto's is_active automatically
};

// ==========================================
// 2. ENCODING (In-Memory -> Binary Bytes)
// ==========================================

// Verify the payload matches the schema rules
const errMsg = UserProfile.verify(userProfile);
if (errMsg) throw Error(errMsg);

// Create a formal Protobuf Message instance, then encode it to raw bytes
const message = UserProfile.create(userProfile);
const encodedBuffer = UserProfile.encode(message).finish();

console.log("--- ENCODING ---");
console.log("Encoded Type:", encodedBuffer.constructor.name); // Uint8Array
console.log("Raw Bytes (Hex):", Buffer.from(encodedBuffer).toString("hex"));
// Output will look like: 082a1204416c65781a0a4a6176615363726970741a0a547970655363726970742001
console.log("Byte Length:", encodedBuffer.length, "bytes");
// Around 34 bytes (JSON equivalent is ~75 bytes!)
```

3. Decoding it

```js
// ==========================================
// --- BINARY BUFFER SENT OVER NETWORK ---
// ==========================================

// ==========================================
// 3. DECODING (Binary Bytes -> In-Memory)
// ==========================================
// The receiver gets the raw Uint8Array and uses the exact same schema to parse it
const decodedMessage = UserProfile.decode(encodedBuffer);

// Convert it back into a plain, native JavaScript object
const decodedProfile = UserProfile.toObject(decodedMessage, {
  longs: String,
  enums: String,
  bytes: String,
});

console.log("\n--- DECODING ---");
console.log("Decoded Name:", decodedProfile.name); // "Alex"
console.log("Decoded Skills:", decodedProfile.skills); // [ 'JavaScript', 'TypeScript' ]
```

## JSON, XML, and Binary Variants

- Moving to standardized encodings that can be written and read by many programming languages (JSON and XML) are the obvious choice
- But, they also have their issues. However, due to their wide adoption, they will likely remain.
- For internal purposes, we might opt for something that's faster to parse and more compact
  - For a small dataset, this doesn't matter much. But, as we get into terabytes of data, the choice of data format becomes more relevant

- Size: xml > json > binary formats

### JSON | Cardbox

Pros:

- human-readable
- lingua france

Cons:

- chatty
- wastes spaces

### XML | Wood box

Pros:

- well-structured

Cons:

- verbose
- usually 2x the size of json
- slow

### Protobuff | High-tech vaccum bag

Pros:

- Very efficient

Cons:

- More complex
- Less readability

### When to use them?

- Need human readability? JSON
- Need performance and evolvability? Protobuf

## Thrift, Protocol Buffers, and Avro

- Binary enconding libs

- Apache Thrift and Protocol Buffers are binary encoding libraries that are based on the same principle as the binary encoding
  - Protocol buffers was originally developed at Google
  - Thrift at Facebook
  - both were made open-source in 2007-08
