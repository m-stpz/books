# Modes of dataflow

So far, in this chapter, we spoke about:

- when we want to send data to another process with which we don't share data with (write to file / send data over the network), we encode it as a sequence of bytes
- different encodings for doing this
- forward and backward compatibility, which are important for evolvability
- compatibility is a relationship between one process that encodes the data, and another that decodes it

In the rest of the chapter, we'll see how data flows between processes:

- via dbs: Dataflow through Databases
- via service calls: Dataflow through Services: REST and RPC
- via async message passing: Message-passing Dataflow

## Dataflow through databases

- Forward compatibility is also necessary here
  - older versions can read newer versions of the code

> Data outlives code

## Dataflow through Services: REST and RPC

- The most common way to organize processes that need to communicate over a network is to have two roles: clients and servers
- Servers: expose an API over the network
  - a server can be a client. E.g., a typical web server acts as a client to a database
- Clients: can connect to the servers and make a request to that API

> API exposed by the server is known as a service

- REST isn't a protocol, but a design philosophy
  - URL + Methods
  - API designed according to the principles of REST is called RESTful

- By contrast, SOAP is an XML-based protocol for making network API requests

### The problems with remote procedure calls (RPCs)

- RPC: Making a request on the network as if it were a function call
- Conceptually, serverless functions and Remote Procedure Calls share the same philosophy:
  - execute a specific function on a remote machine as if it were a local function call, without worrying about the underlying network or server infrastructure
- Inside a cluster where services need to talk to each other with very low-latency, RPCs are great

#### How network requests are different from local execution?

| Feature        | Local                                                  | Network request                                                             |
| -------------- | ------------------------------------------------------ | --------------------------------------------------------------------------- |
| Execution      | Predictable. Either succeeds or fails                  | Unpredictable. Request or response might be lost. A myriad of possibilities |
| Return         | Result/Exception/Infinite loop                         | Result/Exception/Timeout                                                    |
| Idempotency    | Idempotent by default                                  | You need to make it idempotent                                              |
| Execution time | Usually the same                                       | Slower than a function call, also very variable time                        |
| Passing data   | "Simple". Passing the pointers of data in local memory | Data needs to be encoded (in-memory -> byte sequence)                       |

## Message-passing Dataflow

- How does encoded data flow from one process to another in/over `___` ?
  - dbs: one process writes enconded data and another reads it in at some point in the future
  - network: one process sends a request over to the network to another and expects a respose

> Think of message brokers (RabbitMQ, Kafka) as the post. You deliver a letter, continue with you life and they are responsible for delivering it

### Advantages of message broker/queue compared to RPC

- Adds a buffer, so if recipient isn't available or is overloaded, it adds to the queue. Increased reliability
- Redelivers messages to process that crashed. Avoids messages being lost
- One message can be sent to various recipients
- Decouples sender (producer) from recipient (subscriber)

### Message brokers/queues

1. A process sends a message to a named queue/topic
2. Broker ensures the message is delivered to one or more consumers of/subscribers to that queue/topic

- Message brokers typically don't enforce a particular data model. Message = sequence of bytes + metadata
